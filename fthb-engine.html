FTHB AI Research Engine

A Python/FastAPI pipeline that screens fixed-income securities against Biblical Responsible Investing (BRI) criteria — not generic ESG. Output is a FAS score and a human-reviewable decision.

Where It Started

It started as a thin wrapper — a few API calls to Perplexity and Gemini that took whatever came back at face value. No reproducibility, no source-quality rules, no audit trail. Most of the engineering below is the story of turning that into something an investor could actually trust.

The Build Story

Chapter 1
Make it reproducible before making it correct.

Problem: The same input gave different outputs — different enriched descriptions, different source sets, occasionally a flipped Pass→Review decision — and the replay tooling itself couldn't be trusted.

  • Proved the fill step ran at temperature 0, isolating Perplexity's live web search as the drift source.
  • Built a replay harness that caught a field-name bug which was itself flipping decisions.
  • Added an issuer-keyed enrichment cache (sha256 of the canonical input) that produced byte-identical inputs across three runs.
→ You can't fix what you can't observe or reproduce.
Honest framing: enrichment and scoring layers are now deterministic (replay-verified); source retrieval still varies run-to-run by design — which Chapter 2 addresses.
Chapter 2
Don't fight nondeterminism — bound it.

Problem: Perplexity returns a different set of sources every run; you can't force consistency.

Classify every source by tier (primary regulator / secondary press / tertiary / excluded), then enforce evidence rules — critical pillars require a primary source, severe pillars require two independent publishers. Cross-finding deduplication prevents the same event from being double-counted. Doctrine files are hashed into every run for auditability; the loader fails loud rather than silently skipping a missing file.

→ If you can't control which sources arrive, control how they're judged.
Honest framing: 210 passing tests cover deterministic enforcement and regression protection. The test corpus is synthetic by necessity — recent live runs produced few or no scored negative findings.
Chapter 3
The top-8 cap was silently dropping government sources.

Problem: Before sending sources to Gemini, the pipeline kept only the first 8 with no ordering. Because Perplexity's return order is random, an FTC, OSHA, or DOJ source could land at position 9 or later and be silently discarded — Gemini never saw it, and nothing recorded that it had been dropped. A real audit gap.

Fix — two layers:

  • A regulatory safety net that records any primary/regulatory URL that didn't reach adjudication, so nothing disappears silently.
  • Tier-aware ordering that sorts primary → secondary → tertiary before the cap, so regulators reach Gemini first.

Result: regulatory snippets reaching Gemini rose from 8 to 12; sources promoted from beyond the cap rose from 3 to 5.

→ A silently dropped regulator is worse than a wrong answer — at least a wrong answer is visible.
Chapter 4
One upstream change unblocks three rules.

Problem: Three validator rules — a date-provenance check, a per-source entity-mismatch warning, and an anti-bot source substitution — were all blocked on data the pipeline didn't emit.

A single Gemini schema change added event_date_source, source_named_entity, and primary_inaccessible, unblocking all three at once.

→ Find the one upstream change that turns several blocked features into buildable ones.
Honest framing: the date-provenance demotion and per-source entity warning are live; the anti-bot substitution is wired and tested but dormant until the source bundle starts flagging blocked primaries.

Current Frontier In Progress

Chapter 5 — Knowing when it's not a code problem.

The next batch is deliberately blocked on doctrine/policy input, not engineering:

  • Parent/subsidiary attribution — the source_named_entity field is in place to drive the eventual rule, but the rule itself is a judgment call.
  • Pattern-vs-incident distinction.
  • Recency weighting.
  • Pillar tiebreakers.
  • A cleaner reason-code taxonomy.

Two known small open items: apply the Insufficient-Evidence database migration, and close a schema/prompt sync gap on event_date_source. Open

The hard part here isn't writing the code — it's deciding the policy the code should enforce. Shipping the boundary honestly beats faking completeness.

Lessons Learned

1
Observe and reproduce before you optimize — most of the early wins were instrumentation, not features.
2
When you can't remove nondeterminism, build a deterministic layer that bounds its blast radius.
3
Fail loud. A silent-skip loader and a silently truncated source list were the two most dangerous bugs; both hid real data loss behind a green run.
4
Recognize when a problem is a policy decision wearing an engineering costume — and stop, rather than guessing the doctrine.
